## 📌 1. Executive Summary
This document outlines the architecture for an autonomous, multi-turn AI customer support agent designed to resolve common support queries, maintain conversational context, and intelligently escalate out-of-scope issues. The system features a robust Human-in-the-Loop (HITL) mechanism with a **Semantic Propagation Cascade** to auto-resolve similar unanswered queries once an admin provides a solution.

## 💾 3. Data & Storage Layers

To prevent hallucination, maintain context, and power the Admin Dashboard, the system utilizes one relational database for operational metrics and three isolated Vector Databases for semantic search.

### 3.1 SQL Database (Operational & State Tracking)
This database acts as the absolute source of truth for the Admin Analytics Panel and tracks the real-time status of every interaction.
* **What it stores:** The complete interaction log and evaluation state for every user query.
* **Schema Breakdown:**
  * `query_id` (Primary Key): Unique identifier for the specific question.
  * `session_id` (Foreign Key): The thread identifier linking messages to a specific user session.
  * `timestamp`: Exact time of the interaction.
  * `transformed_query`: The optimized, context-rich text string generated by the Transformer Agent.
  * `response`: The text string of the answer (defaults to `"Under Evaluation"` until a successful RAG generation or Admin input).
  * `score`: A float value representing the confidence/accuracy metric evaluated by DeepEval.

### 3.2 Vector DB 1: Answered Queries (Verified Memory)
* **What it stores:** Historically successful, AI-generated answers that passed the DeepEval check, alongside Admin-approved Q&A pairs. 
* **Purpose:** Acts as the primary "memory bank." Before synthesizing a new answer from raw documents, the system checks here to see if a verified, high-quality answer to a similar question already exists.

### 3.3 Vector DB 2: Unanswered Queries (Escalation Queue)
* **What it stores:** User queries that failed the AI evaluation check (score fell below the DeepEval threshold) and require human intervention.
* **Purpose:** This database powers the Admin Dashboard's unresolved ticket queue. It is also the target database for the **Semantic Propagation Cascade**—when an admin answers one question, the system searches this specific Vector DB to find and auto-resolve similar pending queries.

### 3.4 Vector DB 3: Service Documentation (Ground Truth)
* **What it stores:** The core knowledge base, including ingested product documentation, company FAQs, PDF manuals, markdown files, and scraped URLs containing specific details about your services.
* **Purpose:** The factual foundation of the system. The Main RAG Agent searches this database to synthesize novel answers for queries that have not been asked or resolved before.



### 4.2 Detailed Breakdown: Node 3 — Main RAG Agent Node

The Main RAG Agent acts as the central execution engine of the LangGraph workflow. It does not just perform a basic search; it acts as an intelligent aggregator equipped with custom tools to query and cross-reference your isolated storage layers.
